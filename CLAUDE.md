# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

Lagerassistent is a single-page web app (Swedish) for managing an event equipment warehouse. It tracks notes/tasks, physical inventory, returns, and work planning. Deployed on Netlify with a Supabase backend.

## Build pipeline (Fas 2)

This project now uses **Vite + TypeScript** as the build tool (since Fas 2 merge).

- **Source files** live in `src/legacy/*.ts` — classic scripts (not ES modules), shared global scope between files
- **tsc transpiles** `src/legacy/*.ts` → `public/js/*.js` via `tsconfig.legacy.json` (`module: "none"`, `outDir: "public/js"`)
- **Vite** bundles HTML + CSS, serves `public/js/*.js` as static assets, outputs to `dist/`
- Static assets: `public/assets/logo.png` etc.
- `public/js/` is in `.gitignore` (generated)

**Commands:**
- `npm run dev` — runs tsc then Vite dev server on port 5173
- `npm run build` — runs `tsc -p tsconfig.json --noEmit` (modul-config check) then `tsc -p tsconfig.legacy.json` (transpile legacy) then `vite build` → `dist/`
- `npm run typecheck` — TS-check both configs, no emit
- `npm run preview` — serves `dist/` locally

NPM scripts invoke `node node_modules/<bin>` directly to bypass npm cmd-shim's bug with OneDrive paths containing spaces (Windows-specific).

## JavaScript load order

Scripts in `index.html` must remain in this exact order — each file depends on the previous:

1. `/js/config.js` — constants och enums (state ligger nu i store.js)
2. `/js/store.js` — appState + subscribe/notify/optimistic-helper (Fas 4)
3. `/js/supabase.js` — sb/sbPaged/getAuthToken (raw fetch-wrapper)
4. `/js/services/*.js` — per-aggregate CRUD (notes, materials, returns, tasks, info, pins, images)
5. `/js/auth.js` — PIN-login + completeLogin/logout
6. `/js/ui.js` — helpers (modal, toast, filters, AI chat)
7. `/js/render.js` — HTML generation per tab
8. `/js/render/patches.js` — granular DOM-patches (Fas 4.5)
9. `/js/render/subscribers.js` — store-subscribers som triggar cross-cutting patches
10. `/js/realtime.js` — Supabase Realtime WebSocket
11. `/js/components/imageUpload.js` + `commentSystem.js` — generic UI-helpers
12. `/js/actions/*.js` — per-aggregate user-action-handlers

These paths are served by Vite from `public/js/` (in dev) or `dist/js/` (in prod) — both generated by tsc from `src/legacy/`.

## TypeScript model

- `src/domain/types.d.ts` — ambient module-style domain types (Note, Material, Task, etc.). Exported for future ES-module use.
- `src/legacy/types.d.ts` — bridge that re-declares those types globally so legacy files (which can't `import`) see them as ambient.
- `src/legacy/actions/materials.ts` (559 rader) har fortfarande `@ts-nocheck` — den enda kvarvarande efter Fas 4.10. Övriga actions/ är typade.

## Architecture (efter Fas 4)

All mutabel state lever i `src/legacy/store.ts` som `appState` grupperat per domän. Top-level aliases (`auth`, `ui`, `notes`, `materials`, `returns`, `tasks`, `info`, `chat`) ger åtkomst med kort skrivsätt: `notes.list`, `materials.counts[id]`, `auth.user`.

Detta funkar eftersom `tsconfig.legacy.json` har `module: "none"` — alla filer delar global script-scope vid runtime, paritet med klassiska `<script>`-taggar pre-Vite.

**Reactive store-API (Fas 4):**
- `subscribe(key, cb)` / `notify(key)` i `store.ts`. Listeners registreras per StoreKey ("notes" | "materials" | ...).
- `optimistic({ apply, rollback, api, storeKey })` — local-first updates med rollback vid fel. Used för hot-path-status-bytes.

**Dataflow per user-action (typisk):**
1. User triggar funktion i `actions/<aggregate>.ts`
2. Funktionen anropar `services/<aggregate>.ts` (CRUD mot Supabase)
3. Service uppdaterar `appState.<aggregate>.*`
4. Action kör `notify(storeKey)` — subscribers patchar header-meta etc.
5. Action kör `patch<Aggregate>Card(id)` för granular DOM-patch, ELLER `render()` för full re-render (tab-byte, många element ändras)

**Patch-strategi:** `render/patches.ts` exporterar `patchNoteCard/patchMaterialCard/patchTaskRow/patchHeaderMeta`. Hot-paths (`setStatus`, `toggleNote`, `submitComment`, `setTaskStatus`) använder dem för <50ms feedback. Övriga callsites använder fortfarande `render()` — fungerar pga `bindEvents()` re-attaches non-inline listeners.

**Realtime-flow:** `realtime.ts` får postgres_changes → `load*()` → `notify(storeKey)` → patch via subscribers. Inget extra arbete behövs i actions för att andra användares ändringar ska visas.

## Supabase tables

| Table | Purpose |
|---|---|
| `notes` | Warehouse observations; soft-deleted via `deleted_at` |
| `materials_v2` | Material types (two kinds: `is_article_based` and count-based) |
| `material_items` | Individual tracked articles (for article-based materials) |
| `material_counts` | Status counts per material (for count-based materials) |
| `material_history` | Audit log for status changes |
| `borrowed_material` | Rented-in materials from external suppliers |
| `returns` | Returned equipment records |
| `tasks` | Work planning tasks with assignment and status log |
| `task_status_log` | Audit trail for task status changes |
| `user_pins` | bcrypt-hashed PINs (per Fas 1) |
| `comments` | Comments on notes (note: no `updated_at` column — see B-bug in project_todo.md) |

All Supabase calls go through the `sb<T>(path, opts)` wrapper in `supabase.ts`, which adds auth headers using the JWT from sessionStorage (anon key as fallback pre-login).

## Users and roles

Users are hardcoded in `config.ts` (`USERS` array). Role distinction comes from JWT `user_metadata.role` (set by `verify-pin` Edge Function during login). Admin-only features: Export tab, Trash tab, AI chat tab, archived tasks/returns, and setting item status to "tillgänglig".

## XSS safety

All user-generated content inserted into HTML **must** be wrapped with `esc()` (for text nodes) or `escAttr()` (for HTML attribute values). Both helpers are defined in `ui.ts`. Never insert raw user data into innerHTML.

## AI integration

The AI chat and summary features call a Netlify serverless function at `/.netlify/functions/claude`, which proxies requests to the Anthropic API. The function code lives in `netlify/`. The app currently uses `claude-sonnet-4-5` — update this to the latest model when upgrading.

## Material types

Two fundamentally different tracking modes:
- **Count-based** (`is_article_based: false`): Tracks quantities per status (e.g. 50 pall EPS PRO: 40 tillgänglig, 8 uthyrd, 2 reparation). Counts live in `material_counts`.
- **Article-based** (`is_article_based: true`): Each physical item has its own ID and status (e.g. LD20-19). Items live in `material_items`.
